# 0005 — Remaining `var(--...)` references after the @theme migration

## Status

Open. Created during PR-2 of the codebase refactor (Tailwind `@theme` migration).

## Context

PR-2 migrated ~110 `.tsx` files from inline `style={{ color: "var(--text-primary)" }}` and arbitrary-value `bg-[var(--bg-hover)]` patterns to Tailwind v4 utility classes generated by the `@theme` block in [`src/global/styles/theme.css`](../../src/global/styles/theme.css). The codemod handled the simple cases mechanically but skipped patterns it could not safely translate without risking visual regressions.

## What's left

`grep -r "var(--" src --include='*.tsx'` still returns ~140 matches across ~38 files. Every remaining case falls into one of four patterns:

### 1. Ternary expressions inside `style={{}}`

```tsx
style={{
  backgroundColor: isActive ? "var(--btn-primary-bg)" : "var(--bg-secondary)",
  color: isActive ? "var(--btn-primary-text)" : "var(--text-secondary)",
}}
```

Cleanup path: replace with conditional `className`:

```tsx
className={
  isActive
    ? "bg-primary text-primary-foreground"
    : "bg-muted text-muted-foreground"
}
```

### 2. Compound shorthand built from template literals

```tsx
style={{
  border: `1px solid ${error ? "#fca5a5" : "var(--border-default)"}`,
}}
```

Cleanup path: split into separate Tailwind utilities (`border border-border`) plus a conditional override class for the error case.

### 3. Free-floating `CSSProperties` objects defined in module scope

[`src/global/components/SegmentedControl/SegmentedControl.tsx`](../../src/global/components/SegmentedControl/SegmentedControl.tsx) and [`src/global/components/Surface/Surface.tsx`](../../src/global/components/Surface/Surface.tsx) define `CSSProperties` records that are spread into JSX. These need to be refactored into `className` lookups (e.g., a `Record<Tone, string>` mapping `tone` to a Tailwind class).

### 4. Tokens that don't have an `@theme` mapping

A small number of references use colors that aren't part of the design system (e.g., a hardcoded `"#fca5a5"` red for input error borders). These should either be added to the `@theme` block or replaced with the closest existing utility (`destructive-subtle`, etc.).

## Migration

The codemods used during PR-2 lived in `scripts/` and were deleted after the bulk migration. If reviving them for cleanup, the entry points were:

- `migrate-theme-tokens.mjs` — main rewriter (only translated literal `var(--token)` strings and arbitrary `bg-[var(--token)]` classes).
- `strip-var-fallbacks.mjs` — pre-pass that stripped fallback values (`var(--x, fallback)` → `var(--x)`) so the main rewriter could pick them up.
- `fix-duplicate-classnames.mjs` — post-pass that merged duplicate `className=` attributes when the existing class was a template-literal expression.

A second migration pass would need to handle ternaries and template literals in style values. The cleanest approach is per-file manual cleanup since each remaining case has component-specific structure.

## Tracking

When touching any file with leftover `var(--...)` references for unrelated work, migrate the references in that file as part of the same change. Avoid introducing new `var(--...)` references in `.tsx`; use the `@theme`-generated utilities instead.
